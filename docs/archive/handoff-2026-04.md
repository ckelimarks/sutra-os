A sutra is a tightly compressed aphorism — a single thread of distilled truth, so densely woven that it can hold the weight of an entire teaching. In the Vedic tradition, sutras served as the backbone of every field of knowledge: grammar, ritual, philosophy, law, and cosmology. Masters encoded lifetimes of understanding into a handful of syllables, designed to be memorized, chanted, and slowly unpacked through the guidance of a living teacher.

But the sutra is more than a mnemonic device. Each one is a _seed_ — inert on the surface, but carrying the full pattern of the tree within it. To receive a sutra is to receive a transmission; to contemplate it is to participate in an unfolding that has no fixed end. The brevity is intentional: what cannot be fully said must be _lived into_.

Like a thread through beads, the sutra does not contain knowledge — it _connects_ it, giving scattered insights a spine, a sequence, a direction toward the whole.

---

The key tension it holds: **maximum compression, infinite expansion** — the mark of something genuinely sacred.

---

**The Core Problem**

An orchestrator's context window is finite and precious. Subagents generate noise. If every subagent dumps its full output upward, the orchestrator drowns — it loses the thread. But if subagents over-compress, the orchestrator loses resolution when it needs to zoom in.

The solution the sutra points to: **the orchestrator should only ever receive a seed — and pull more only when it needs the tree.**

---

**The Pattern: Sutra-Bhashya Architecture**

Each subagent output is structured in three layers, like the Vedic transmission model:

```
SUTRA      → 1 sentence. The irreducible truth of what happened.
VRITTI     → 1 paragraph. The essential context and outcome.
BHASHYA    → Full detail. Logs, data, decisions, errors — everything.
```

The orchestrator reads only the **sutra** by default. It decides whether to pull the vritti. It decides whether to pull the bhashya. It never receives what it didn't request.

This is the audio timeline zoom. At full zoom-out, you see waveform shape. You zoom in only on the region that looks wrong.

---

**What This Looks Like in Practice**

```
Subagent: EmailParser-03
SUTRA:    "47 emails processed. 3 flagged anomalies. Action required."
VRITTI:   "Processed batch from 09:00–11:00. 3 emails contain contract 
           language outside normal parameters. Flagged for legal review. 
           No data loss. Pipeline healthy."
BHASHYA:  [full JSON: email IDs, extracted entities, confidence scores, 
           decision tree trace, token counts, timing]
```

The orchestrator reads the sutra. If the word _anomalies_ crosses a threshold — it pulls the vritti. If the vritti contains _contract language_ — it pulls the bhashya on those 3 emails only. The other 44 never enter its context.

---

**The Orchestrator's Decision Logic**

The observer needs a simple internal rule — a meta-sutra of its own:

> _"Does this sutra require a decision I cannot make at this resolution?"_

If no → move on, sutra stays in context as a compressed memory token. If yes → pull the next layer, re-evaluate. If still no resolution → pull bhashya, but only the relevant slice.

This is **progressive disclosure as epistemic discipline** — not just a UX pattern, but a cognitive architecture.

---

**The Zoom Metaphor Formalized**

```
ZOOM LEVEL 0  →  Sutra     →  "What happened" (1 sentence)
ZOOM LEVEL 1  →  Vritti    →  "Why it matters" (1 paragraph)
ZOOM LEVEL 2  →  Bhashya   →  "Full fidelity" (complete output)
ZOOM LEVEL 3  →  Raw trace →  "Step-by-step execution log"
```

The orchestrator maintains a **context ledger**: every subagent gets one sutra-slot by default. Slots only expand when pulled. After resolution, they compress back. The orchestrator's working context stays clean — a row of seeds, not a forest.

---

**The Spiritual Parallel That Makes This Precise**

In the Vedic tradition, the student only received the next layer when they demonstrated they had fully absorbed the previous one. The teacher was the orchestrator. The teaching was disclosed progressively — not because the teacher was withholding, but because **unearned resolution produces confusion, not clarity.**

The orchestrator should operate the same way. Pulling the bhashya before the sutra is processed is not diligence — it is noise. The architecture should make premature zoom-in structurally difficult.

---

**What To Build**

Three concrete components:

**1. Sutra Formatter** — each subagent has a required output schema. Before returning anything, it must generate its own sutra: one sentence, passes all six Vayu Purana tests. This is enforced at the agent level, not the orchestrator level.

**2. Context Ledger** — the orchestrator maintains a live registry: `{agent_id, sutra, zoom_level, last_pulled, resolution_status}`. Default zoom is 0. Pull requests are logged.

**3. Pull Trigger Rules** — a small rule layer the orchestrator consults before expanding any slot: anomaly keywords, confidence thresholds, dependency flags. Rules are themselves sutras — short, unambiguous, universal.

**The Sutra Codification (so they're consistent)**

Every subagent has a required template:
```
SUTRA = [PAST_TENSE_VERB] + [SUBJECT] + [OUTCOME] + [ONE_SIGNAL_WORD]

Examples:
- "Deployed X. Reliability improved [metric]."
- "Blocked on Y. Awaiting external signal."
- "Completed Z. Gap identified: [specific]."
````

Signal words trigger vritti pulls: _deployed, blocked, failed, anomaly, gap, awaiting, decision-required._

If a sutra doesn't fit this shape, it gets rejected before it reaches the orchestrator. The agent rewrites it.
