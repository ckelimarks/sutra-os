# OSS Sync Workflow

Sutra has two repositories:
- **`sutra-build/`** — canonical source (local, full context, internal docs)
- **`sutra-os/`** — public OSS (sanitized, no internal planning, no personal paths)

The OSS repo is a **derived artifact**, not an independent source. This document defines the workflow.

---

## Principle

> Do not edit `sutra-os` directly. It is a build artifact, like `dist/` in a JS project. Treat it as generated, not authored.

All work happens in `sutra-build`. Changes to `sutra-os` come only from sanitization.

---

## Workflow 1: Push local changes to OSS

When you've shipped a feature or fix in `sutra-build`:

```bash
cd Projects/prototypes/sutra-build
./sanitize-and-push.sh
```

This:
1. Runs `/sanitize-for-oss` skill (strips internal docs, personal paths, agent names)
2. Diffs the sanitized version against current `sutra-os`
3. Shows you what changed
4. Applies changes + commits + pushes to GitHub

**Confirm the diff before pushing.** The script is interactive by default.

For CI/CD or batch updates:
```bash
./sanitize-and-push.sh --force
```

---

## Workflow 2: Address an OSS issue/PR

Someone files a bug or PR on GitHub. Steps:

1. **Read the issue on GitHub.** Understand what's broken.
2. **Reproduce in `sutra-build` if possible.** You have full context; use it.
3. **Fix in `sutra-build`.** With full context, internal docs, agent names, everything.
4. **Test locally in `sutra-build`.** Verify the fix works.
5. **Commit to `sutra-build`** with a clear message.
6. **Re-sanitize and push:**
   ```bash
   ./sanitize-and-push.sh
   ```
7. **Close the OSS issue** with a reference to your `sutra-build` commit:
   > "Fixed in sutra-build commit abc123def. Changes synced to public repo."

Never edit the bug-fix code twice (once in OSS, once in local). Fix once in the source, sanitize, push.

---

## Workflow 3: OSS PR comes in

Someone submits a PR directly to `sutra-os`. Steps:

1. **Review the PR.** Look at the diff.
2. **Understand the intent.** What problem does it solve?
3. **Cherry-pick the equivalent change into `sutra-build`.** Not a blind copy-paste — adapt it to the canonical source with full context.
4. **Test in `sutra-build`.** Verify it works with the full codebase.
5. **Commit to `sutra-build`.** With a message like: "Merge OSS PR #42: [description]. Original: github.com/[org]/sutra-os/pull/42"
6. **Re-sanitize and push** to close the OSS PR:
   ```bash
   ./sanitize-and-push.sh
   ```

The OSS PR gets updated automatically. Close it with a message explaining you merged the intent into the canonical source.

## Sanitization rules

The `sanitize-for-oss` skill removes:

- **Internal-only docs:** CLAUDE.md, ARCHITECTURE.md, internal strategy sections of ROADMAP.md
- **Personal paths:** `<user>`, `project`
- **Agent names:** WorkerAgent → WorkerAgent, Sutra → Orchestrator (generic terminology)
- **Inline examples with personal context:** References to personal projects, internal URLs

What it keeps:

- **Code** (server/, web/)
- **Public-facing docs:** README.md, API.md, INFRA.md, SESSION-JSONL.md
- **Feature/design docs:** PRD.md, ROADMAP.md (public version)
- **Architecture for OSS:** API contracts, data flow, integration points

## Checklist: before your first push to OSS

- [ ] All docs consolidated into `sutra-build/docs/`
- [ ] ARCHITECTURE.md exists and is marked internal-only
- [ ] CONSOLIDATION-PLAN.md documents the merge (audit trail)
- [ ] `sanitize-and-push.sh` is executable and tested locally
- [ ] `sutra-os/` exists locally at `~/Downloads/sutra-os/`
- [ ] `sutra-os/` has a remote pointing to GitHub
- [ ] Run `./sanitize-and-push.sh`, review the diff, confirm it looks right
- [ ] Test a local fix → sanitize → verify in sutra-os (manual test)

---

## Anti-patterns (don't do these)

- ❌ Edit `sutra-os` directly and push. It will be overwritten on next sync.
- ❌ File a bug in `sutra-os`, fix in `sutra-os`, forget to port to `sutra-build`. The fix will disappear on next sync.
- ❌ Merge OSS PRs directly to `sutra-os` without evaluating in `sutra-build` context. You lose the full picture.
- ❌ Skip the diff review. Always know what's changing before pushing.

---

## Questions?

- **"Why two repos?"** Because public OSS and internal development have different scopes. Sanitized version is smaller, safer to share.
- **"What if I need to fix something in OSS but don't want to disrupt sutra-build?"** Fix it in sutra-build in a feature branch. Test. Merge to main. Then sync. (Still one-way.)
- **"What if OSS gets more active than sutra-build?"** The best problem to have. Shepherd PRs into sutra-build, port them thoughtfully. OSS is the feedback loop.
